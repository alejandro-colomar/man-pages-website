Patches for man-pages

If you know how to fix a problem in a man page (if not, look here), then send a patch, and make sure that you:

  • Put mtk.manpages@gmail.com in "To:". (Please do not CC my @man7.org email address.)
  • Put linux-man@vger.kernel.org into "CC:" (no HTML mail please),
  • Include the string "[patch]" in the subject line

The above is the minimum needed to ensure that there's some chance I'll see the patch. (If you did that, and I do not respond, then ping the mail thread, "replying to all". Unfortunately, I get more mail than I can sometimes deal with, and things do get lost.)

To make your patch even more useful, please note the following points:

  • Make sure that the mail has a suitable Subject line (i.e., one that mentions the name(s) of the page(s) being patched). A suitable subject line might be something like: 
        [patch] shmop.2: Add "(void *)" cast to RETURN VALUE
  • For nontrivial patches:

    • Describe how you obtained the information in your patch: was it by reading (or writing) the relevant kernel or (g)libc source code; by writing a test program (send it with the patch, if you want, and if it is clear and simple to use); from a standards document; from other documentation; from a mailing list or online forum (please provide a URL if possible)?

    • Where relevant, include source code comments that cite commit hashes for relevant kernel or glibc changes: 
            .\" commit <40-character-git-hash>
      Doing so can be useful to help verify details of the patch, or for later historical reference.

    • Please attempt to find and CC relevant developers. If your patch documents a substantial feature or change, and you know which developers added the feature or made the change that your patch documents, please CC them on the patch; with luck they may review and fix your patch. If you don't know who the developers are, you may be able to discover that information from mailing list archives or from Git logs or logs in other version control systems. Obviously, if you are the developer of the feature being documented in a man-pages patch, please identify yourself as such.

    • Please consider CCing relevant mailing lists. If your patch documents a substantial feature or change, please consider CCing relevant mailing lists on your patch. With luck, subscribers on the mailing list might review your patch. Relevant mailing lists may include:

      • linux-kernel@vger.kernel.org for patches to system call pages
      • netdev@vger.kernel.org for patches to pages relating to networking and sockets

      The GNU C library developers don't consider man-pages to be any sort of official documentation for the library. However, if you have a new page for Section 3 or a substantial addition to one of those pages, you might consider CCing libc-alpha@sourceware.org; there are some friendly folk there who may review or comment.

  • For trivial patches, I use a number of keywords that are processed by scripts that automatically build the release changelogs. Please employ the following in the patch subject line, where appropriate:

    • ffix: Formatting fix.
    • wfix: Minor wording fix.
    • tfix: Typo fix.
    • srcfix: Change to man page source that doesn't affect formatted output.
    • spfix: Spelling fix.
    • wsfix: White-space fix.

    Obviously, there's some overlap between the above cases; just choose the one that you think fits best. Using this scheme, a patch's subject line might look like: 
        [patch] tcp.7: tfix
  • Some general tips on how to make my life easier:

    • Send logically separate patches (e.g., for unrelated pages, or for logically separate issues in the same page) as separate mails.

    • Patches that contain correctly formatted groff are preferred. The basics of groff are pretty simple, and you can probably learn as much as you need by just looking at the source of the page to be patched; see also the groff_man(7) and man-pages(7) man pages. If you are uncertain of groff, then send a patch that uses some plain text.

    • Make patches against the latest version of the man page. For information on downloading the latest tarball, or pulling from the man-pages Git repository, see the download page.
      If the man page that you want to patch is not in the tarball or Git repository, that's because it comes from a package other than man-pages; in that case, you should look here.
    • In the body of the mail message, identify the man-pages version against which the patch applies.

    • Send patches in diff -u format; inline inside the mail message is usually best. If it is a very long patch, then send it both inline and as an attachment. Since the man-pages project uses Git, you may find it useful to employ git-send-email or git-format-patch.

  • Some things that you don't need to do:

    • Don't try to patch the COLOPHON section of a page in one of the release tarballs. That text is autogenerated as part of the release process.

    • You don't need to update the page timestamp (".TH" line). My release scripts do that automatically.
